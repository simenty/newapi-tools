### 原始需求摘要

在 V3.1 基础上，分四批交付：
- **第一批 A+D**：CLI 体验增强（交互式安装向导 + config chmod + mirror 并发测试 + 命令别名）及自动更新/版本管理
- **第二批 B**：日志诊断升级（audit list + 错误码文档 + doctor --verbose）
- **第三批 C**：多实例管理（instance 子命令 + status --instance）
- **第四批 E**：文档社区（安装教程、错误码参考页、CHANGELOG 标准化）

---

## 二、产品目标

1. **降低安装门槛**：通过完整的交互式安装向导，让零基础用户无需手写配置文件即可完成 newapi 的初始化部署。
2. **提升运维自主性**：提供自动更新、版本回滚、审计查询等能力，让用户在无技术支持的情况下完成日常运维。
3. **支持多环境管理**：通过多实例命令，让进阶用户在同一台机器上管理测试/生产等多套 newapi 环境。

---

## 三、用户故事

### 第一批：CLI 体验增强（A）+ 自动更新（D）

1. As a **零基础用户**，I want **运行 `newapi install --interactive` 时被逐步引导填写端口、域名、数据库等选项**，so that **无需理解 docker-compose.yml 语法也能完成部署**。
2. As a **运维用户**，I want **`newapi mirror test` 显示每个镜像的延迟并自动推荐最快的**，so that **我能快速切换到最优镜像源，避免拉取镜像超时**。
3. As a **任意用户**，I want **运行 `newapi update --check` 即可知道是否有新版本**，so that **我不必手动去 GitHub 查看 Release**。
4. As a **任意用户**，I want **运行 `newapi update --self` 自动下载并替换二进制，同时备份旧版本**，so that **升级过程安全且可回滚**。

### 第二批：日志诊断升级（B）

5. As a **运维用户**，I want **`newapi audit list --last 20 --cmd install` 过滤查看审计日志**，so that **快速定位某次安装操作的历史记录**。
6. As a **开发者**，I want **`newapi doctor --verbose` 显示每项检查的具体检测细节**，so that **排查环境问题时有足够信息定位根因**。

### 第三批：多实例管理（C）

7. As a **进阶用户**，I want **`newapi instance add/list/switch/remove` 管理多套环境**，so that **在同一机器上隔离测试环境和生产环境**。
8. As a **进阶用户**，I want **`newapi status --instance prod` 查看指定实例状态**，so that **无需手动切换上下文即可监控任意实例**。

### 第四批：文档社区（E）

9. As a **新用户**，I want **getting-started.md 包含 --interactive 安装的完整截图与说明**，so that **首次使用时有图文参考**。
10. As a **开发者**，I want **错误码参考页从代码自动生成**，so that **文档与代码保持同步，不出现错误码与说明不一致的问题**。

## 四、需求池

### P0 — 必须交付（V3.2 核心，第一批 A+D）

| # | 需求 | 所属模块 | 说明 |
|---|---|---|---|
| A-01 | `install --interactive` 完整交互向导 | install | 引导端口/域名/MySQL or SQLite/Redis 开关/镜像加速；生成 .env + docker-compose.yml |
| A-02 | `config chmod` 自动修复（0644→600） | config/doctor | doctor 发现权限异常时自动修复，消除 WARN |
| A-03 | `mirror test` 并发延迟测试 + 排序 | mirror | 并发 HTTP HEAD 所有内置源，展示延迟表格，标记最快源 |
| A-04 | 命令别名：`newapi ls` = `newapi status --all` | root cmd | 在 Cobra 中注册别名 |
| D-01 | `update --check`：对比 GitHub Release 最新 tag | update | 调用 GitHub API 或解析 releases/latest，输出版本对比 |
| D-02 | `update --self`：下载最新二进制替换自身 | update | 下载 → 校验 SHA256 → 备份旧版本 → 替换 → 提示重启 |
| D-03 | 更新前自动触发 backup | update | `update --self` 执行前调用 backup 逻辑 |

### P1 — 应该交付（第二批 B + 第三批 C 基础）

| # | 需求 | 所属模块 | 说明 |
|---|---|---|---|
| B-01 | `audit list` 命令 | audit | 支持 `--last N`、`--cmd <name>`、`--since <date>` 过滤，表格输出 |
| B-02 | 错误码文档自动生成 | apperr/docs | 从 apperr 包注释生成 Markdown 错误码参考表，集成到 `make docs` |
| B-03 | `doctor --verbose` 每项检查详情 | doctor | 逐项输出检测依据（文件路径、命令输出、期望值 vs 实际值） |
| C-01 | `instance add/list/switch/remove` | instance | 实例元数据存 ~/.config/newapi-tools/instances.json；switch 切换 current 指针 |
| C-02 | `status --instance <name>` | status | 读取对应实例配置，复用现有 status 逻辑 |

### P2 — 可以交付（第四批 E）

| # | 需求 | 所属模块 | 说明 |
|---|---|---|---|
| E-01 | getting-started.md 更新（--interactive 教程） | docs | 含步骤截图/录制 asciinema |
| E-02 | 错误码参考页（面向用户） | docs | 与 B-02 产物对齐，发布到 mkdocs site |
| E-03 | CHANGELOG.md 标准化（Keep a Changelog 格式） | docs | 补全 V3.0/3.1 条目，V3.2 新增条目 |

## 六、待确认问题

| # | 问题 | 影响范围 | 建议 |
|---|---|---|---|
| Q1 | `update --self` 在 `/usr/local/bin` 下需要 sudo，如何处理权限提升？ | D-02 | 检测安装路径权限，必要时提示用户加 `sudo`，或提供 install-to-$HOME 选项 |
| Q2 | `instance` 的实例隔离粒度：是否各实例独立的 docker network + volume 命名空间？还是共用同一个 Compose project？ | C-01/C-02 | 建议独立 project name（`COMPOSE_PROJECT_NAME=newapi-<instance>`），避免端口冲突 |
| Q3 | `mirror test` 并发数上限？网络受限环境（如代理）是否需要特殊处理？ | A-03 | 建议并发数可配置，默认 6，超时 3s |
| Q4 | `audit list` 输出格式是否需要支持 `--json` 以便管道处理？ | B-01 | 建议与 `status --json` 保持一致，加 `--json` 标志 |
| Q5 | 错误码文档自动生成：触发时机是 `make docs` 还是 CI 流程？生成的 Markdown 提交到仓库还是只在 site 里？ | B-02/E-02 | 建议生成到 `docs/errors.md`，提交到仓库，mkdocs 引用 |
| Q6 | V3.2 是否计划支持 ARM64（如树莓派、Apple M 系芯片的 Linux VM）？影响 D-02 的下载 artifact 选择逻辑 | D-02 | 建议 `update --self` 通过 `GOARCH` 自动匹配下载链接 |
